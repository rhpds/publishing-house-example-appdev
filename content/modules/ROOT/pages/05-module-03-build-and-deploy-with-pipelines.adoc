You have made changes to the Postcard Generator. Now it is time to ship them.

OpenShift Pipelines (Tekton) provides Kubernetes-native CI/CD. You push code to your Git repository, and the pipeline automatically clones, builds, tests, and deploys the updated application. No Jenkins server. No external CI service. Everything runs on the same cluster as your app.

In this lab, you will push a code change and watch the pipeline do the rest.

== Learning objectives

By the end of this lab, you will be able to:

* Trigger an automated CI/CD pipeline by pushing a code change to Git
* Monitor a Tekton pipeline run through the OpenShift console
* Verify that a pipeline-deployed change is live in the running application
* Describe how OpenShift Pipelines provides Kubernetes-native CI/CD

== Exercise 1: Make a code change

You will add a new destination to the Postcard Generator using the Gitea web editor. This simulates a developer pushing a feature change that triggers the pipeline.

. Open your Gitea instance in a browser:
+
{gitea_url}

. Sign in with your credentials:
+
* Username: `{gitea_user}`
* Password: `{gitea_password}`

. Navigate to your `postcard-generator` repository.

. Click on the `data` directory, then open `destinations.json`.

. Click the *pencil icon* (Edit) in the top-right corner of the file view.

. Add a new destination to the JSON array. Make it something memorable so you will recognize it after deployment:
+
[source,json]
----
{
  "name": "Tokyo",
  "country": "Japan",
  "lat": 35.6762,
  "lng": 139.6503,
  "tagline": "Greetings from Tokyo -- Where Tradition Meets Tomorrow!",
  "colors": {
    "primary": "#c0392b",
    "accent": "#f5b7b1"
  }
}
----
+
NOTE: Make sure to add a comma after the previous entry in the JSON array.

. Scroll down, enter a commit message (for example, "Add Tokyo destination"), and click *Commit Changes*.

The commit triggers a webhook to OpenShift Pipelines. The pipeline starts automatically.

== Exercise 2: Watch the pipeline run

The pipeline is already running. Let's watch it.

. Switch to the OpenShift Console:
+
{openshift_console_url}

. Ensure you are in the *Developer* perspective (dropdown in the top-left corner).

. Select your namespace from the project dropdown: `{user}-postcard-generator`

. Click *Pipelines* in the left navigation.

. Locate the `postcard-generator` pipeline. A new *PipelineRun* should be in progress, shown with a blue running indicator.

. Click into the PipelineRun to see the visual pipeline graph.
+
The pipeline executes these tasks in sequence:
+
* *git-clone* -- fetches the latest code from Gitea
* *build* -- builds a new container image using the updated source
* *test* -- runs the application test suite
* *deploy* -- updates the running deployment with the new image

. Click on individual tasks to view their logs.
+
Watch the build task compile the application, the test task validate it, and the deploy task push the new image to the internal registry and roll out the updated deployment.

. Wait for the pipeline to complete. All tasks should show green checkmarks.
+
This typically takes 3 to 5 minutes.

=== Verify

Run the following to confirm the pipeline completed successfully:

[source,role="execute"]
----
oc get pipelinerun -n {user}-postcard-generator --sort-by=.metadata.creationTimestamp -o jsonpath='{.items[-1].status.conditions[0].reason}'
----

Expected output:

----
Succeeded
----

* All pipeline tasks show green checkmarks in the console
* The PipelineRun status is *Succeeded*

== Exercise 3: Verify your change is live

The pipeline deployed the updated application. Let's confirm.

. Open the Postcard Generator application:
+
{postcard_app_url}

. Click *Generate Postcard* repeatedly until your new destination (Tokyo) appears.
+
The postcard shows Tokyo with a red-toned color scheme, the map pin on Japan, and your tagline. This change was pushed through the pipeline, built into a new container image, and deployed automatically.

// TODO: Add screenshot of the Tokyo postcard
image::postcard-tokyo-deployed.png[Postcard Generator showing the Tokyo destination deployed via pipeline,link=self,window=blank,width=700,title="Tokyo postcard, deployed via pipeline"]

[start=3]
. Return to the OpenShift Console and explore the pipeline in more detail.

. Click on *Pipelines* in the left navigation, then click the pipeline name (not the run) to see the pipeline definition.
+
Notice how each task is defined declaratively. The pipeline is a Kubernetes resource, versioned in Git, running as pods on the cluster.

. Check the deployment to see the new container image tag:
+
[source,role="execute"]
----
oc get deployment postcard-generator -n {user}-postcard-generator -o jsonpath='{.spec.template.spec.containers[0].image}'
----
+
The image tag corresponds to the pipeline run that just completed.

. View your PipelineRun history:
+
[source,role="execute"]
----
oc get pipelinerun -n {user}-postcard-generator --sort-by=.metadata.creationTimestamp
----
+
Each row represents a pipeline execution with its status and duration.

=== Verify

* The Tokyo postcard appears in the Postcard Generator
* The deployment is running the new container image
* The PipelineRun history shows a successful run

== Learning outcomes

By completing this lab, you should now understand:

* ✓ How OpenShift Pipelines automates the build-test-deploy cycle with a single `git push`
* ✓ That pipelines are Kubernetes-native, defined as YAML, versioned in Git, and running as pods on the cluster
* ✓ How a standardized pipeline lets developers focus on code while the platform handles the rest
* ✓ That every app in the organization can ship the same way: consistent, auditable, and repeatable

== Module summary

You pushed a code change and watched OpenShift Pipelines build, test, and deploy it automatically. From commit to running app, the entire process was hands-off.

*What you accomplished:*

* Made a code change in Gitea that triggered the pipeline via webhook
* Watched the Tekton pipeline clone, build, test, and deploy the updated app
* Verified the new destination appears in the live Postcard Generator
* Explored the pipeline structure and run history

*Why this matters:*

* Developers do not need to understand pipeline internals to benefit from them. Push code and it ships.
* The platform team defines the pipeline once. Every app uses the same process.
* Pipelines are auditable. Every change has a traceable build history.
