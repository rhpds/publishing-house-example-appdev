Your Postcard Generator is deployed and running. But what happens when someone makes a manual change on the cluster? Scales the pods, tweaks a setting, adjusts a resource limit?

OpenShift GitOps (Argo CD) watches your deployment and compares it against the desired state defined in Git. If anything drifts, Argo CD catches it and can sync the deployment back to what Git says it should be.

In this lab, you will intentionally break things and watch Argo CD fix them.

== Learning objectives

By the end of this lab, you will be able to:

* Navigate the Argo CD dashboard and inspect an application's resource tree
* Intentionally cause configuration drift on the cluster
* Observe Argo CD detect drift and report an OutOfSync status
* Sync an application to restore the Git-defined desired state

== Exercise 1: Explore Argo CD

Argo CD is the GitOps engine that keeps your cluster in sync with Git. Let's see how it views the Postcard Generator.

. Open the Argo CD dashboard:
+
{argocd_url}

. Click *Log in via OpenShift* and use your lab credentials:
+
* Username: `{user}`
* Password: `{password}`
+
NOTE: You may be asked to authorize read-only access to your user information. Click *Allow selected permissions* if prompted.

. On the Argo CD dashboard, locate the `postcard-generator` application.
+
It should show *Synced* (green) and *Healthy* (green heart icon). This means the cluster state matches what Git defines.

. Click into the application to see the *resource tree*.
+
This is a visual map of every Kubernetes resource that makes up the Postcard Generator: Deployment, Service, Route, ConfigMap, and more. Each node shows its current status.

. Click the *Deployment* node in the tree.
+
In the details panel, note:
+
* Current replica count
* Container image
* Last sync timestamp
* The Git commit SHA that this deployment matches

== Exercise 2: Break it -- Cause drift

Now you will intentionally make a manual change on the cluster that contradicts what Git says. This simulates someone making an ad-hoc change, the kind of thing that causes configuration surprises in production.

. Switch to the OpenShift Console:
+
{openshift_console_url}

. Ensure you are in the *Developer* perspective and select your namespace: `{user}-postcard-generator`

. Open the *Topology* view and click on the Postcard Generator deployment.

. In the *Details* panel on the right, find the pod count. Click the *up arrow* (^) next to the pod count to scale the deployment from 1 pod to 3 pods.

. Wait a few seconds for the additional pods to start. You now have 3 pods running, but Git says there should be 1.
+
You just created drift.

. Switch back to the Argo CD dashboard.
+
Within about a minute, the application status changes from *Synced* to *OutOfSync*. The Deployment node in the resource tree turns yellow.

. Click the *Deployment* node. Argo CD highlights the difference: `spec.replicas` is 3 on the cluster but 1 in Git.

. Click *Diff* on the Deployment to see the exact difference between the live cluster state and the Git-defined state.
+
This is how you catch unauthorized or accidental changes. No guessing, no auditing logs. Argo CD shows you exactly what drifted and when.

== Exercise 3: Fix it -- Sync and verify

Argo CD detected the drift. Now let's sync the application back to the desired state.

. In the Argo CD dashboard, with the Postcard Generator application open, click *Sync* at the top of the application view.

. In the sync dialog, review what will be changed. The Deployment replicas will be set back to 1.

. Click *Synchronize* to apply the correction.

. Watch the sync progress. Argo CD reverts the Deployment to match Git. The status returns to *Synced* (green).

. Switch to the OpenShift Console and check the Topology view.
+
The pod count is back to 1. Argo CD restored the Git-defined state.

. Confirm via the terminal:
+
[source,role="execute"]
----
oc get deployment postcard-generator -n {user}-postcard-generator -o jsonpath='{.spec.replicas}'
----
+
Expected output:
+
----
1
----

Now let's make a *legitimate* change, one that goes through Git.

[start=7]
. Open your Gitea instance:
+
{gitea_url}

. Navigate to the `postcard-generator-deploy` repository (the deployment manifests repo, separate from the application source).

. Open the Deployment YAML file and click the *pencil icon* to edit.

. Add an environment variable to the container spec:
+
[source,yaml]
----
env:
  - name: POSTCARD_GREETING
    value: "Wish you were here!"
----

. Commit the change with a descriptive message.

. Return to Argo CD. The application shows *OutOfSync* again, but this time the diff shows the new environment variable you just added through Git.

. Click *Sync*, then *Synchronize* to apply the change.
+
The application updates and the new environment variable is applied to the running Postcard Generator.

== Learning outcomes

By completing this lab, you should now understand:

* ✓ How GitOps uses Git as the single source of truth for deployment configuration
* ✓ How Argo CD continuously monitors for drift between desired and actual cluster state
* ✓ That manual cluster changes are detected and can be reverted, preventing configuration surprises
* ✓ How legitimate changes flow through Git, creating an audit trail and enabling code review for infrastructure

== Module summary

You intentionally broke your deployment, watched Argo CD catch the drift, and synced it back to the correct state. Then you made a legitimate change through Git and watched Argo CD apply it properly.

*What you accomplished:*

* Explored the Argo CD dashboard and resource tree
* Caused configuration drift by manually scaling pods
* Observed Argo CD detect and report the drift with an exact diff
* Synced the application to restore the Git-defined state
* Made a proper change through Git and deployed it via GitOps

*Why this matters:*

* GitOps means no more "who changed that?" mysteries. Every change is tracked in Git.
* Argo CD prevents configuration drift from becoming a production incident.
* The same workflow works across clusters, regions, and clouds. Git is the universal control plane.
