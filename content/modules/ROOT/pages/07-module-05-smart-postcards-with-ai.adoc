The Postcard Generator works great, but its taglines are canned. "Greetings from Paris!" is fine. It is also the same every time.

What if the app could write something original for each postcard? Something tailored to the destination, creative, and different every time?

OpenShift AI provides model-as-a-service (MaaS), letting your application call an LLM endpoint with a simple REST API. No model hosting. No GPU management. No ML expertise. Just send a prompt, get a response.

In this lab, you will flip a switch and watch the Postcard Generator go from canned taglines to AI-crafted messages.

== Learning objectives

By the end of this lab, you will be able to:

* Locate a model-as-a-service endpoint in OpenShift AI
* Enable an AI feature in an application using environment variables and Kubernetes Secrets
* Observe an application consuming an LLM endpoint to generate dynamic content
* Describe how MaaS lowers the barrier to integrating AI into any application

== Exercise 1: Explore the MaaS endpoint

Before enabling AI in the app, let's see what is available.

. Open the OpenShift Console:
+
{openshift_console_url}

. Click the *application launcher* (grid icon) in the top navigation bar and select *OpenShift AI* (or navigate directly to the OpenShift AI dashboard if available).

. In the OpenShift AI dashboard, navigate to *Model Serving*.
+
Locate the available model endpoint. Note the model name and the endpoint URL. This is a text-generation LLM ready to receive prompts via a standard REST API.
+
This endpoint is already provisioned and running. No model deployment, no GPU configuration, no training required. It is AI as a service.

== Exercise 2: Enable AI-powered postcards

The Postcard Generator already has the AI integration code built in. It is just turned off by default. The app uses canned taglines until an environment variable tells it to call the MaaS endpoint instead.

. In the OpenShift Console, navigate to your Postcard Generator Deployment:
+
*Topology* -> click the Postcard Generator node -> *Details*

. Click *Actions* (top-right) -> *Edit Deployment* (or go to the *Environment* tab if available).

. Add a new environment variable:
+
* *Name*: `POSTCARD_AI_ENABLED`
* *Value*: `true`
+
Alternatively, set it via the terminal:
+
[source,role="execute"]
----
oc set env deployment/postcard-generator POSTCARD_AI_ENABLED=true -n {user}-postcard-generator
----

. Save the change. The deployment rolls out a new pod with the AI feature enabled.

. Wait for the rollout to complete:
+
[source,role="execute"]
----
oc rollout status deployment/postcard-generator -n {user}-postcard-generator
----
+
Expected output:
+
----
deployment "postcard-generator" successfully rolled out
----

The Postcard Generator is now wired to the MaaS endpoint. Every time it generates a postcard, it sends the destination name and country to the LLM and gets back a creative, personalized message.

== Exercise 3: See it in action

Time for the fun part.

. Open the Postcard Generator:
+
{postcard_app_url}

. Enter your name in the *From* field.

. Click *Generate Postcard*.
+
Instead of a canned tagline like "Greetings from Paris!", you should see a unique, AI-generated message. Something like:
+
____
Dear friend, the northern lights of Reykjavik are calling your name. Bundle up and bring your sense of wonder -- this land of glaciers and geysers has stories only the brave get to hear.
____

. Generate several more postcards. Each message is different, tailored to the destination, and creative in a way that canned text never is.

[start=5]
. Now let's look under the hood. Open your Gitea instance:
+
{gitea_url}

. Navigate to the `postcard-generator` repository and open `app/ai_client.py`.
+
Review the key parts:
+
* *Prompt template*: The app sends the destination name and country to the LLM with a request for a creative, travel-themed postcard greeting
* *API key handling*: The key is read from the `postcard-ai-config` Kubernetes Secret via an environment variable. Never hardcoded.
* *Fallback behavior*: If the AI endpoint is unavailable or slow, the app falls back to canned taglines. AI enhances but never breaks the app.

. Finally, let's see the difference side by side. Disable the AI feature:
+
[source,role="execute"]
----
oc set env deployment/postcard-generator POSTCARD_AI_ENABLED=false -n {user}-postcard-generator
----

. Wait for the rollout:
+
[source,role="execute"]
----
oc rollout status deployment/postcard-generator -n {user}-postcard-generator
----

. Generate a postcard. You are back to canned taglines.
+
The difference is immediately visible. Re-enable AI when you are done exploring:
+
[source,role="execute"]
----
oc set env deployment/postcard-generator POSTCARD_AI_ENABLED=true -n {user}-postcard-generator
----

=== Verify

* AI-generated postcards produce unique, destination-specific messages
* Disabling AI returns the app to canned taglines
* Re-enabling AI restores the dynamic messages

== Learning outcomes

By completing this lab, you should now understand:

* ✓ How OpenShift AI provides model-as-a-service, making AI consumption as simple as calling a REST API
* ✓ That integrating AI into an existing application is an environment variable and a Kubernetes Secret, not a rewrite
* ✓ How the fallback pattern (AI when available, static when not) keeps the app reliable
* ✓ That platform teams provision the AI endpoint and credentials while developers simply consume them, the same pattern as a database or message queue
